标准工作流程

在处理项目时，自然会创建精灵，编辑脚本，删除时间表以及随着项目的发展而需要执行的其他任何操作。这些都是您可能要跟踪并在出现问题时找回的所有操作。这是使用“源代码管理”的主要原因-可以还原任何事故-我们将迅速从头开始引导您完成基本工作流程，以便您了解其工作原理。

从头开始，创建一个新的GML项目，然后通过main选项激活源代码控制。这使我们可以访问顶部的“源代码管理”菜单，因此我们将立即选择“创建项目存储库” 。如果查看房间资源，您会看到默认房间出现了红色惊叹号，表示它已从其最后一个状态被修改。如果我们在项目中还有其他资源，您会看到它们都有相同的图标，如下图所示：

现在，我们可以选择按原样提交空白项目，也可以使其变为基本状态。现在，我们将进行第一次提交，因此请单击“源代码控制” >“提交更改” 。现在，我们将打开一个新窗口，该窗口分为三个区域：

  • 分阶段的更改：这表示将提交给源的更改。这可能为空白，但可能会显示Main Options（主选项）更改，这很好。
  • Unstaged Changes ：显示已更改的文件，但是我们没有告诉Git我们要提交它们-它们正在“挂起”更改。
  • 提交消息：这是我们可以添加的注释，以说明所做的更改。

现在，因为我们要提交所有更改，请单击“全部登台”按钮，然后输入“首次提交！”之类的内容。到“提交消息”中，然后单击“提交” 。现在我们所有的资源都应该有一个绿色的勾 ，表示未找到任何修改。

值得注意的是，当你这个窗口内工作，各种演出和不分阶段的消息将通过一个符号前缀。使用的符号将根据操作和文件而有所不同，并使用以下符号：

  • [...] -分阶段变更（以前是[A]）
  • [+] -新增加
  • [->]-重命名/移动
  • [-]-已移除
  • [M]-非阶段性变更
  • [?]-未跟踪的文件
  • [!]-文件丢失

现在可以开始我们的工作流程，并且可以开始构建项目并保存并提交更改，就像在上面的段落中所做的一样。

工作一段时间后，您可能需要检查并查看在任何给定时间所做的事情，因此，您需要打开提交历史记录。要打开历史记录窗口，您只需转到“源代码管理” >“查看历史记录”即可打开另一个包含三个窗格的窗口：

顶部窗格描述提交，中间窗格描述提供给所选提交的注释，底部窗格描述在此提交中已修改的文件。请注意，如果您已经设置了Diff Tool ，则在此窗口中双击任何文件将打开diff工具，您将能够看到文件之间的更改。
